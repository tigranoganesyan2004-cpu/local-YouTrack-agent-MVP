# Локальный MVP агента по задачам YouTrack

## Что это
Консольный локальный агент, который работает **сначала по выгрузке XLSX/CSV**, а потом без смены архитектуры может быть расширен:
1. до прямой загрузки данных из YouTrack REST API;
2. до загрузки вложений по задачам;
3. до отдельного контура дообучения/LoRA.

## Почему архитектура именно такая
Текущая выгрузка по сути описывает **контур согласования документов**, а не классический backlog разработки:
- статусы — это в основном состояния согласования;
- есть поля по согласованию с ДИТ / ГКУ / ДКП / ДЭПиР;
- есть сроки согласования;
- есть тип документа, инициатор, ответственный и описание.

Поэтому MVP должен отвечать не только на вопрос «что за задача», но и на вопросы:
- на каком этапе согласования задача;
- у кого она сейчас;
- какие сроки есть;
- какие похожие согласования уже были;
- были ли похожие кейсы с замечаниями / отказами / отменой.

## Что умеет MVP
- загрузка XLSX/CSV;
- приведение строк к единой схеме;
- поиск по ID задачи;
- фильтры по статусу, приоритету, типу документа, функциональному заказчику, ответственному (ДИТ);
- поиск похожих задач (hybrid: lexical + embeddings + RRF);
- режим анализа новой формулировки;
- режим дедлайнов по согласованию;
- локальная история запросов в SQLite;
- LLM используется только как **слой формулировки ответа по уже найденным данным**;
- при ошибке LLM есть детерминированный fallback без фантазии.

## Ограничения ноутбука
Ноутбук с i5-1135G7 и 16 ГБ ОЗУ подходит для:
- Python + pandas + SQLite;
- локального индекса на сотнях/тысячах задач;
- Ollama с маленькими моделями уровня 1.5B–3B;
- embeddings на текущем объеме данных.

Ноутбук **не подходит как комфортная машина** для:
- полноценного обучения модели;
- нормального fine-tuning 3B/7B на PyTorch;
- тяжелой обработки больших пакетов вложений;
- многопользовательского сервиса.

## Реальный план развития
### Этап 1. MVP по файлам (делаем сейчас)
Источник данных: XLSX/CSV из выгрузки.
Результат: локальный консольный агент.

### Этап 2. Прямая интеграция с YouTrack
Источник: REST API.
Добавляем:
- импорт issues;
- custom fields;
- links;
- attachments metadata;
- comments metadata;
- локальный кэш в JSON/SQLite.

### Этап 3. Работа с вложениями
Добавляем:
- скачивание attachments;
- извлечение текста;
- отдельный индекс по вложениям;
- ответ: сначала issue, потом attachment evidence.

### Этап 4. Дообучение
Выносится в отдельный контур.
Не смешивать с MVP.
Для этого этапа уже стоит просить более мощную машину.

## Что просить по мощностям
### Можно не просить пока
Для MVP и последующей интеграции с API этого ноутбука хватит.

### Стоит просить позже
Когда пойдут:
- массовые вложения;
- OCR/PDF parsing;
- PyTorch/LoRA;
- 7B+ модели;
- несколько пользователей.

Минимально разумный апгрейд под этап обучения:
- 32–64 ГБ RAM;
- NVIDIA GPU с 16–24 ГБ VRAM;
- SSD 1 ТБ.

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
copy .env.example .env

```

## Папки
- `data/raw` — исходные выгрузки;
- `data/processed` — подготовленные JSON;
- `data/index` — векторный индекс;
- `data/history` — SQLite история;
- `src/youtrack_api.py` — будущая прямая интеграция;
- `src/attachment_ingest.py` — будущий контур вложений.

## Почему обучение не включено в MVP
Пока задача — сделать **рабочий, проверяемый, нефантазирующий retrieval-агент**.
Если начать с обучения сразу:
- резко вырастет сложность;
- станет труднее дебажить;
- будет сложно понять, где ошибка: в данных, retrieval, prompt или fine-tuning.

Правильный порядок:
1. retrieval по файлам;
2. API;
3. attachments;
4. только потом обучение.
