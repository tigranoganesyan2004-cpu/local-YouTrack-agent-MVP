# local-YouTrack-agent-MVP

Локальный аналитический агент для работы с выгрузкой задач YouTrack.

Проект предназначен для быстрого поиска задач, нахождения похожих кейсов, проверки дедлайнов, фильтрации по полям и получения структурированного ответа по найденному контексту. Агент работает локально, использует Ollama для генерации и embeddings, FAISS для семантического поиска, SQLite для хранения истории и FastAPI для локального web-интерфейса.

---

## Что делает агент

Текущий pipeline проекта:

1. Загружает выгрузку из `data/raw/` (`.xlsx` и/или `.csv`)
2. Нормализует данные и сохраняет:
   - `data/processed/tasks.json`
   - `data/processed/dataset_report.json`
3. Строит FAISS-индекс:
   - `data/index/tasks.index`
   - `data/index/tasks_mapping.json`
4. Принимает запрос пользователя
5. Определяет режим обработки
6. Выполняет retrieval
7. При необходимости вызывает LLM только по найденному контексту
8. Возвращает структурированный ответ
9. Сохраняет историю в SQLite:
   - `data/history/agent_history.db`

---

## Основная архитектура

Проект построен по принципу **retrieval-first**:

1. сначала подготавливаются данные;
2. затем строится индекс;
3. потом пользователь задает запрос;
4. агент определяет режим;
5. сначала ищутся релевантные задачи;
6. только после retrieval может вызываться LLM;
7. LLM не должна фантазировать вне найденного контекста.

Это важно, потому что проект не является “просто чат-ботом”.  
Это локальный аналитический инструмент по задачам YouTrack.

## Основные возможности

### Подготовка данных
- чтение `XLSX` / `CSV`;
- маппинг колонок во внутреннюю схему;
- нормализация дат, статусов, текстов;
- формирование:
  - `semantic_text`
  - `metadata_text`
- сохранение нормализованного датасета и краткого отчета.

### Retrieval
- поиск по ID задачи;
- exact / deterministic search;
- lexical search;
- semantic search;
- hybrid search через RRF;
- cross-encoder re-ranking поверх hybrid results;
- фильтрация по полям;
- подсчеты по полям;
- дедлайны на N дней.

### LLM-логика
- retrieval + LLM только по найденному контексту;
- structured JSON output;
- fallback при плохом ответе модели;
- валидация ответа перед финальной выдачей.

### История
SQLite история сохраняет:
- время запроса;
- режим;
- текст запроса;
- итоговый ответ;
- найденные issue_id;
- длительность;
- использовалась ли LLM;
- текст ошибки, если была.

### Интерфейсы
- CLI через `main.py`
- локальный web UI через FastAPI

## Режимы работы

Проект поддерживает три пользовательских режима web-интерфейса:

### 1. Auto
Система сама определяет сценарий обработки.

### 2. Exact
Детерминированный поиск без LLM.  
Подходит для:
- воспроизводимых результатов;
- отладки retrieval;
- быстрых точных запросов.

### 3. LLM
Retrieval + LLM.  
Подходит для:
- поиска похожих задач;
- анализа новой постановки;
- свободных аналитических вопросов.

## Intent detection

В проекте используется двухуровневая схема определения намерения пользователя:

### 1. Детерминированный слой
Работает мгновенно и без LLM:
- `help / помощь`
- `ид / show / id / показать`
- `точно / exact`
- `похожие / similar`
- `анализ / analyze / проанализируй`
- `список / list`
- `количество / count`
- `сроки / deadlines`
- а также явный `issue_id` в свободном тексте

### 2. LLM-классификация
Если детерминированный слой не сработал:
- используется короткий prompt;
- Ollama возвращает JSON с mode / query / filters / field / days;
- результат кэшируется в памяти;
- при любой ошибке идет fallback в `general_search`.

## Re-ranking

После hybrid retrieval проект может делать второй проход через cross-encoder:

- модель: `cross-encoder/ms-marco-MiniLM-L-6-v2`
- библиотека: `sentence-transformers`

### Зачем это нужно
Lexical и semantic retrieval выбирают кандидатов, но финальный порядок не всегда идеален.  
CrossEncoder смотрит на пару:

- запрос
- `task["semantic_text"]`

и более точно переранжирует top-кандидатов.
